# 输出模板

这是默认的输出形态。它规定了**形式上的契约**——哪些层面必须出现、用什么表示法、在什么粒度上不能糊弄过去；它不规定**内容上的细节**——每节具体写什么、按什么顺序展开、是否合并相邻两节，应该由当前论文语料和你想讲清楚的那条线决定。

把这份模板当成一份"必须覆盖到的层面"的清单和一组"具体度锚点"，而不是一份硬性的填空表。如果某节按引导自然写出来已经比下面提到的还更具体、更连贯，就不要再被本文档里的提问拉回去——本文档存在的意义是让分析不浮空，而不是让分析按 bullet 拼起来。

## 全局写作原则

- **逻辑先行，结构其次。** 优先把一段话讲连贯，让读者跟得上因果链；不要为了对齐模板而把一个完整的判断切成五个互不连通的 bullet。
- **具体度高于齐整。** 当你写到"场景"、"瓶颈"、"insight"、"技术动作"这类抽象词时，立即问自己：能不能再具体一档？能不能从"机器人操作"走到"长程多步骤厨房任务里跨房间记忆失效"这种粒度？如果不能，要么补，要么说明为什么不能。
- **判断 + 展开。** 重要判断之后通常要补一段展开（用括号、用副句、或单独一段都可以），让读者看明白这句话具体什么意思、证据来自哪、为什么这个点对后续判断重要。
- **时间是隐含坐标轴。** 论文目录里通常带发表日期，脚本会把它们解析为 `Published hint`。在你的判断里，应该明显能看到时间线在起作用：哪些 framing 是早期建立的、哪些是被后来推翻或修正的、哪个时间点是拐点、当前还在演化中的是什么。不需要每节都重复时间线，但你的判断不能像把所有论文摊在同一个时间点上一样。
- **点名具体论文。** 重要判断后面要点出对应论文标题，不要靠"有些工作"、"近期论文"这种说法把事情糊过去。

## 1. 领域前景与时间线

**目的：** 让一个完全没接触过这个方向的人，看完这一节就大致知道这个方向在解决什么问题、为什么这件事重要、当前演化到什么阶段、为什么这批论文能代表它的一段轨迹。

**值得展开的角度：** 这个方向最初为什么出现？早期的 framing 长什么样？后来哪些瓶颈让它转向？现在前沿在往哪边推？这批论文里时间最早的几篇和最近的几篇之间，到底发生了什么实质性的变化？时间不是必须按年表写，但读者读完应该能感受到"这件事是一步步长出来的"，而不是"这是一堆并列的论文"。

**深度锚点：** 与其写"这个方向研究机器人操作的世界模型"，不如写"早期世界模型主要在 Atari 这类离散环境上验证'未来预测能力是否能转换为决策能力'的 framing；2023 之后随着 VLA 兴起，研究重心转向'连续动作 + 长程任务下的世界模型如何接动作接口'；这批论文里最早的 X 还沿用前者的预测式接口，而最近的 Y 已经把动作直接出在生成图像上"。后者让人看到了一条线，前者只是一句标签。

**形式：** 中文为主，专业术语可保留原文。这一节写给新手，不要假设他理解 jargon。

## 2. 源论文锚点

**目的：** 把整批 follow-up 的"原点"指出来，让后面所有论文都可以参照它定位。

**值得展开的角度：** 哪一篇是源论文（如果根目录下有同名 markdown 文件，默认就是它）？它在自己那个时间点上提出了什么 framing 或机制？它当时面对的是什么状况，所以才"必须"提出这个东西？为什么后来的工作会围着它打转，而不是绕过它？它哪些假设是后来 follow-up 接受的、哪些是被改写的？

把源论文放回它自己的时间坐标里讲——它是 2022 vs 2024 的源论文，意味很不一样。

**形式：** 必须以 `[[源论文标题]]` 形式 Wiki Link 一次（这样后续的图谱和 backlinks 才能串起来）。

## 3. 重点论文的 Challenge / Insight 分析

**目的：** 让读者读完每篇重点论文的分析，能用自己的话复述"这篇在干什么、为什么这件事难、它选了哪条路、留下了什么"。这一节是后面所有 idea 的根基——它薄一寸，后面 idea 就虚一尺。

**值得展开的角度：**

- 它真正在和什么"难"较劲？不是 abstract 里那句口号，而是它实际想破的那个机制层面或评测层面的难点。
- 这个难点在什么场景里暴露出来？任务形态、数据来源、操作环境、评测设置——具体到一个新人能想象出"这个失败的瞬间长什么样"。
- 它选的是哪条路？换的是模型架构、训练目标、表征接口、监督信号、推理流程，还是评测设定本身？
- 它留下来的 insight 是什么？是那种一年甚至更久之后还会被引用、还会影响别人怎么 frame 问题的东西，而不是"我们把 X 提了 3 个点"。
- 它做了什么取舍、付出了什么代价、依赖什么前提？这些常常比 contribution 部分更能告诉你这篇论文的真实位置。
- 它在时间线上落在哪？是建立在更早的某个 framing 上做延伸，还是在反驳某个流行假设，还是在解决前一波工作集中暴露出来的瓶颈？

**深度锚点：** "它解决长程任务规划"——这是浮的；"它发现现有 VLA 在跨房间任务里因为视觉 memory 被推理 token 挤占而出现物体遗忘，并把 memory 拆成了一个独立的可被显式查询的 cache"——这是落地的。哪怕你只能写到中间那一档，也至少要离前一种远、离后一种近。

**形式：** 表格、bullet、短段落、混排都可以——选最让逻辑流动的那种。可以在论文标题位置直接用 `[[论文标题]]`，方便后面图谱沿用。证据要指向（哪段摘要、哪段引言、哪个实验或对比），不要凭印象。

## 4. 这批论文真正带来的技术进步

**目的：** 在 challenge 分析的基础上"抬一层"——不是逐篇说"它做了 X"，而是站在整批论文之上回答："这一年（或几年）这个方向的能力边界往前挪了哪几格？"

**值得展开的角度：** 哪些进步是机制层面的（模型真的能做以前做不了的事）？哪些是接口层面的（同样的能力被接到了能落地的地方）？哪些是评测层面的（终于有了能撑住强 baseline 的测法）？哪些其实只是表述变化、系统堆叠、或者把别人的成功包装一层？哪些"进步"主要靠基础模型变强带飞，而不是这条线自己的贡献？时间线在这里特别有用——把进步"挂"到具体时间点上，能让读者看见拐点。

**深度锚点：** 不要写"planning 能力变强了"。要写到例如"早期 VLA 只能做单步动作预测，2024 中之后的 X、Y、Z 三篇通过把 chain-of-thought 显式化到 token 流里，第一次让长程任务的成功率从 ~20% 拉到 ~60%；但这条路依赖大模型 backbone，所以同期 W 提出了一个不同接口……"——具体到时间、能力变化幅度、所依赖的条件、与之并行的另一条路。

**形式：** 不要硬凑固定分类。归出几条真的发生了的进步，每一条都有明确的证据指向（哪些论文、哪些对比、哪些现象），并明确区分实质性进步与表面更新。

## 5. 还没有解决的关键问题

**目的：** 在你看完整批论文之后，把仍然没被解决的问题留给读者——而且让他知道为什么这些问题难、为什么会卡住后续工作、为什么值得继续追问。

**值得展开的角度：** 哪些问题在多篇论文里反复出现、被多次声称要解决但实际上都只是缓解了？哪些问题在最新的论文里仍然是失败模式的主因？哪些问题被绕开了（论文换了一个更窄的评测，避开它）而不是解决了？哪些问题随时间变得更难还是更易？

**深度锚点：** 不要列"sample efficiency"、"generalization"这种通用 bottleneck——它们正确但没有信息量。要落到例如"长程任务里跨场景的 object permanence 仍然没解决，X 和 Y 都声称做了，但它们的评测都把场景切换限制在 3 帧以内；真正长跨度场景下的失败率仍未被报告"——这样读者立刻知道"原来这件事还没真做"。

**形式：** 每个未解决问题后面都补论证（在哪些论文里如何暴露、为什么还没解决、它接下来会卡住什么样的工作）。

## 6. 值得复用的 Technical Insight

**目的：** 把这批论文里"值得装进自己工具箱"的技术模式抽出来——不是一两个 trick，而是那种 framing 或机制级别的、未来做相邻方向时还能拿来用的东西。

**值得展开的角度：** 哪些 insight 是这批论文里被多次独立验证的？哪些 insight 已经被证明在某些场景能放大杠杆、但在另一些场景失败？哪些 insight 看起来是"通用模式"但其实有强适用边界？哪些被某篇论文埋了一笔但没充分挖掘、值得后续 idea 拣起来？

**深度锚点：** "把动作 head 替换成视频生成"是 insight；"通过 attention sink 的方法稳定长 context 的视觉 memory"是 insight；"用更大模型"不是 insight。每条 insight 后面要说清楚：哪些论文支持它、它的适用边界在哪、为什么对后续工作重要。

**形式：** 每条 insight 都用一句概括 + 一段展开。不要堆术语。

## 7. follow-up 脉络图

**目的：** 用一张可以直接粘进 Obsidian 的链接图，把这批论文之间的真实关系结构显出来。

**值得展开的角度：** 谁在谁的基础上做延伸？谁在反驳谁？谁和谁面对的是同一个瓶颈但走不同的机制路径？谁和谁的模块互补、可以拼成一个完整方案？哪些 follow-up 之间的横向连接，比它们各自回连源论文更重要？

**形式约束（这一节是硬要求）：**

- 优先用兼容 Obsidian 的关系行表示法，例如：
  - `[[源论文]] -> [[后续论文A]] : 开出这条分支`
  - `[[Paper A]] -> [[Paper B]] : 在它基础上延伸`
  - `[[Paper C]] -> [[Paper D]] : 修正 / 质疑`
  - `[[Paper E]] -> [[Paper F]] : 同一瓶颈，不同机制`
  - `[[Paper G]] <-> [[Paper H]] : 同一场景，不同技术路线`
  - `[[Paper I]] <-> [[Paper J]] : 技术模块互补`
- 不能写成"所有论文都只指回源论文"的星型结构——只要有合理依据，就要补 follow-up 之间的横向连接。
- 在人能读懂的关系图后面，再生成一份机器可读关系清单，默认保存到 `$PWD/.paper_followup_links.md`。这份清单只保留标准关系行，每行一条，中间不要夹杂解释文字——它要喂给脚本。
- 推荐紧接着运行：

  ```bash
  python3 scripts/update_obsidian_links.py \
    --root "$PWD" \
    --relations "$PWD/.paper_followup_links.md"
  ```

  这一步把双链真正写回每篇论文 note，Obsidian 才会有真实 backlinks。

- 如果 Wiki Link 图谱不够直观，可以再补一张 `mermaid` 作为辅助可视化，但 Wiki Link 那份是主，mermaid 是辅。
- 图后面要用一段文字解释这张图——它真正揭示了什么聚类？哪些链接最 load-bearing？时间线在这张图里有没有形成可见的脉络？这段解释要配证据。

## 8. 论文分级

**目的：** 把"哪些论文研究信号强、哪些弱"讲清楚——本质是研究信号判断，不是价值判断。源论文不要混进 follow-up 排名里，单独说。

**值得展开的角度：** 这篇论文的 framing 是不是真的对准了领域瓶颈？它的机制是不是真的改变了能力边界，还是主要靠 scaling？它的证据强度是否撑得起它的 claim？它的 ablation 能不能解释为什么有效？它留下来的东西是 incremental 还是 foundational？

**深度锚点：** "高信号：这篇论文 framing 清晰、机制有效"——这是空话。要写到例如"高信号：这篇论文是这一批里第一次把动作生成接到图像生成 pipeline 上，机制不是简单替换 head 而是重写了训练目标；它的能力提升不能用 scaling 解释；ablation 把每一个改动都拆出来对比过；后续多篇论文沿用了它的接口"——这样分级才有信息量。

**形式：** 默认 `高信号 / 中信号 / 低信号` 三档，但如果当前语料更适合别的标签（例如"foundational / incremental / 旁支"），就换。每篇论文给一句它做了什么、一句解释为什么落在这一档、必要时再加一句新人应该从它身上学到什么。

## 9. 二十个候选研究 idea

**目的：** 把前面所有 challenge / insight / 未解问题真正"用起来"，撑开一个广的可能性空间——这是后续筛选的原料。

**值得展开的角度：** 哪些 insight 还没被充分挖掘，可以拣起来？哪些瓶颈在多个分支里反复出现，值得做一个跨分支的解？哪些场景被这批论文集体回避了，可能是下一个金矿？哪些在时间线上"早期被绕开"的问题，现在条件已经成熟可以重做？哪些 idea 看似异想天开但实际上能在小规模快速试出来？

**这一阶段的硬要求：**

- **必须凑齐 20 个**——少了就发散不够，没法支撑后面的筛选阶段。
- **风格允许差异很大**——一两个稳妥延伸式 idea、几个对准明确瓶颈的 grounded idea、几个长程或激进 idea，是健康的混合。
- **不要按预设分类配额来凑**——例如"必须有一个 benchmark idea、一个 representation idea"这种是反模式。idea 应该从论文里真正暴露的问题自然长出来。
- **不要写"提升 planning"、"更好的 representation"这种没有机制的占位 idea**——这种 idea 在阶段 B 一定会被自己淘汰，写出来只是污染样本。

**深度锚点：** 一个合格的 idea 写完之后，一个新人应该能用一段话复述"它在打什么 challenge、它的技术假设是什么、它要改的是模型/目标/数据/监督/评测里的哪一块、第一组最小实验大概怎么搭、什么结果算成功"。这五件事不必排成五条 bullet——一段连贯的论述往往比五条 bullet 更有说服力，关键是看完之后这个 idea 不"飘"。

**形式：** 每个 idea 至少要有标题 + 一段足够具体的论述。鼓励引用前面 challenge / insight 那一节的对应论文（用 `[[论文标题]]`），让 idea 和证据基础显式连起来。

## 10. 第二轮筛选

**目的：** 在 20 个 idea 上面再做一层判断，让真正有研究杠杆的浮上来，让只看起来新的沉下去。

**值得展开的角度：** 哪些 idea 真正对准了反复出现的瓶颈？哪些其实已经被某篇论文做掉了一半（前面 challenge 分析里应该已经显出来）？哪些依赖不现实的数据/算力/硬件？哪些可以在一两周做出第一个信号？哪些更像 thesis 方向，需要长期投入？哪些其实只是包装更新，没改变能力边界？

**这一节不要"客气"。** 每个被淘汰的 idea 都要说清它倒在哪一关——这是这一节的全部价值。如果 20 个 idea 你保留 18 个写"都不错"，等于没做这一步。

**形式：** `靠谱 / 存疑 / 不靠谱` 是默认标签，但如果当前 idea 集分布更适合另一种边界（例如"短期可发表 / 长期方向 / 学习型项目 / 可暂缓"），就换。每个判断后面都要补论证——光给标签等于没筛。

## 11. 所有"靠谱"档 idea 的完整展开

**目的：** 把第 10 节里**所有**判定为"靠谱"的 idea 全部展开到能开工的程度。这一节**不再挑出"最推荐的几个"**——筛选已经在第 10 节做完了，凡是被留在靠谱档的，就都值得读者据此动手。重要的是详细，不是精简。

**为什么不再挑"最推荐"几个**：研究方向的"最优"高度依赖读者的兴趣、组里的资源、当前手头能调动的人和算力——这些信息你（写综述的人）大多不掌握。强行收敛到 1-3 个反而让读者看不到其他靠谱选项的具体度，损失信息。把所有靠谱 idea 都写透，让读者根据自己的现实情况选，是更负责任的做法。

**每个靠谱 idea 都要展开到下面这些维度**（不必排成固定字段，让它流动成连贯论述）：

- 它值得做的根本原因——对准的是哪个具体瓶颈，那个瓶颈的证据来自哪几篇论文
- 它要建立在哪些 prior paper 上——用 `[[论文标题]]` 显式连起来
- 第一组实验怎么设计才能在最短时间拿到一个能"判生死"的信号
- 什么结果算成功，什么结果算失败
- 最大的失败模式是什么——评测设计塌了？机制没起作用？数据不够？baseline 太强？
- 即便失败，能留下什么有价值的副产品（数据集？工具？某个 negative result？）
- 实施周期的粗估——是 2-4 周能跑出第一个信号，还是需要数月的 infrastructure 搭建？这件事很影响读者的决策

**深度锚点**：每个 idea 写完后，一个学生应该能据此排出未来 2-4 周的实验日程。如果某个 idea 写不到这个具体度——说明它其实不是"靠谱"，应该回到第 10 节降级为"存疑"。换句话说：**这一节的密度反过来检验第 10 节的判断**——写不出 4 周日程的 idea 等于没真的判定它靠谱。

**形式**：每个 idea 单独一个三级标题（`### Idea N: ...` 或 `### 方向 N: ...`），按 idea 顺序展开。不要把内容压缩成短词条，让逻辑流动。每一点关键判断后面都配论证（建立在哪几篇论文的判断上、依据是什么、为什么这样设计能 cover 这个考量）。

**数量预期**：靠谱档常见数量在 3-7 个之间。如果筛出 1 个或 0 个，可能是第 10 节过严；如果超过 10 个，可能是第 10 节过松。这种异常值得在第 10 节回去检查，但不强制改——也许这批论文确实留下了大量可做空间，或者确实没多少。

## 12. 用户自带 Idea 评估

**目的：** 如果用户提供了自己的 idea（包括"靖文的这个 idea 怎么样"这类请求），把它放回前面整张图里独立评估。如果用户没提供，明确写一句"用户暂未提供自带 idea"，不要省略这一节。

**值得展开的角度：** 这个 idea 对准的是不是这批论文反复暴露出来的关键瓶颈，还是一个表面上听起来重要但实际上不是这条线主战场的问题？哪些论文是它的同盟（论据/方法/数据上）、哪些论文让它变得可疑或多余？它的新意落在哪一层（问题 framing？机制？数据/监督？评测？）——并且诚实指出哪几层其实已被现有工作覆盖。最便宜的验证路径是什么——一个能在一两周内做完、能让用户决定是否继续投入的实验。

**形式：** 最后给一个清楚的判断（默认 `靠谱 / 存疑 / 不靠谱`，可换更合适的标签）。判断必须配上完整的论证，包括把它推向这个标签的具体论文证据。这一节是用户最在意的部分——不要简写。

## 13. Reading 与行动建议

**目的：** 给一个新手看完这份分析之后，知道接下来一周、一个月、一学期分别该做什么。

**值得展开的角度：** 在这批论文里，先精读哪几篇？为什么是这几篇——是因为它们是 framing 起点、还是机制最干净的实例、还是对推荐方向最直接的 prior？哪些论文只需要快读了解 framing 就够了？先复现哪一个工作最划算——它要既容易跑通、又能让人摸到这条线的核心机制？还缺什么背景或工具技能？现在先不要砸两周时间在哪些事上（即使它们看起来很有意思）？

**深度锚点：** "先读 X 和 Y"——这是空建议。要写到例如"先精读 X，因为它的 framing 是这条线的源头，后面所有论文都假设你懂它的 setup；然后快读 Y、Z、W 了解三种主流接口；先复现 V 的最小版本，因为它是这批论文里 codebase 最干净、又能让你触到核心 trade-off 的；先不要碰 U——它的复现成本极高，而且对你接下来的方向没有直接帮助"——这样新人离动手只差一步。

**形式：** 每条建议都配上为什么（建立在前面哪几节的判断上）。

## 形式约束总览（硬要求）

- 默认全部用中文输出，论文标题、模型名、数据集名、专业术语可以保留原文。
- 重要判断后面要点出对应论文标题，不能用"有些工作"、"近期论文"这类说法糊过去。
- 第 7 节的 Wiki Link 关系图与机器可读关系清单不能省。
- 第 9 节必须凑齐 20 个 idea，第 10 节必须做实质筛选。
- 当结论主要建立在 digest、摘要和引言上而非全文，要明确说"证据来自摘要 / 引言层面，置信度 X"。
- 区分"论文声称"和"你的判断"——这两件事在文字上不能混。
- 整体口吻：要求高、对不确定性诚实、不写 hype 文案、不写冷冰冰百科条目，像一个真正在帮学生选题、避坑、起步的人。
